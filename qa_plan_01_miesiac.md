### Dzień 14 (alt) — Page Object Model głębiej + Factory + Builder

> 📌 *POM to nie katalog metod — to wzorzec z regułami. Factory i Builder eliminują magiczne dane w testach.*

**🎯 Cel bloku**: Piszesz test z POM + Factory dla danych testowych

🕐 **BLOK 1 (45 min)** — POM rules + Factory pattern
- Materiał: [Page Object Model — Martin Fowler](https://martinfowler.com/bliki/PageObject.html) (darmowy, kanoniczny)
- Zasady POM których często się nie przestrzega:
  - Page Object NIE zawiera asercji (to rola testu)
  - Page Object zwraca inny Page Object po nawigacji
  - Jeden Page Object = jedna strona lub komponent
- **Factory pattern** dla danych testowych:
  ```python
  # Zamiast: user = {"name": "test123", "email": "test@test.com"}
  # Używaj:
  class UserFactory:
      @staticmethod
      def standard() -> dict:
          return {"name": "Jan Kowalski", "email": "jan@example.com", "role": "user"}
      @staticmethod
      def admin() -> dict:
          return {"name": "Admin", "email": "admin@example.com", "role": "admin"}
  ```
- Ćwiczenie: napisz `UserFactory` i `ProductFactory` dla saucedemo.com

☕ **PRZERWA (15 min)**

🕑 **BLOK 2 (45 min)** — Builder pattern dla API requests
- Kiedy używać: gdy request body ma 5+ opcjonalnych pól
  ```python
  # Zamiast: requests.post(url, json={"field1": "x", "field2": None, "field3": True, ...})
  # Używaj:
  class CreateUserRequest:
      def __init__(self): self._data = {}
      def with_name(self, name): self._data["name"] = name; return self
      def with_role(self, role): self._data["role"] = role; return self
      def build(self): return self._data
  # Użycie: CreateUserRequest().with_name("Jan").with_role("admin").build()
  ```
- Materiał: [Builder pattern Python](https://refactoring.guru/design-patterns/builder/python/example) (darmowy)
- Zadanie: napisz Builder dla jednego endpointu reqres.in

📝 **PODSUMOWANIE (15 min)**

### Dzień 15 (alt) — Singleton, Strategy, wzorce CI

> 📌 *Singleton kontroluje zasoby (driver, sesja DB); Strategy pozwala testować na różnych środowiskach*

**🎯 Cel bloku**: Rozumiesz kiedy użyć Singleton i Strategy w frameworku testowym

🕐 **BLOK 1 (45 min)** — Singleton + Strategy
- **Singleton** — jedna instancja w całej suicie:
  ```python
  # Konfiguracja środowiska — tylko jedna instancja
  class Config:
      _instance = None
      def __new__(cls):
          if not cls._instance:
              cls._instance = super().__new__(cls)
              cls._instance.base_url = os.getenv("BASE_URL", "https://staging.example.com")
          return cls._instance
  ```
  ⚠️ Uwaga: driver NIE powinien być Singleton (problemy z równoległością) — Config/Settings TAK
- **Strategy** — podmiana zachowania bez zmiany testu:
  ```python
  # Testuj na różnych przeglądarkach tym samym testem
  class BrowserStrategy(ABC):
      def create_driver(self): ...
  class ChromeStrategy(BrowserStrategy): ...
  class FirefoxStrategy(BrowserStrategy): ...
  # fixture czyta env var i wybiera strategię
  ```
- Materiał: [Strategy pattern](https://refactoring.guru/design-patterns/strategy/python/example) (darmowy)

☕ **PRZERWA (15 min)**

🕑 **BLOK 2 (45 min)** — Decorator pattern (test reporting + retry)
- **Decorator** w testach = opakowanie funkcji bez zmiany jej kodu:
  ```python
  # Automatyczny retry dla flaky tests
  def retry(times=3):
      def decorator(func):
          def wrapper(*args, **kwargs):
              for i in range(times):
                  try: return func(*args, **kwargs)
                  except AssertionError:
                      if i == times - 1: raise
          return wrapper
      return decorator
  ```
- Ćwiczenie: napisz `@retry(3)` decorator i `@screenshot_on_fail` decorator
- Materiał: [Python Decorators](https://realpython.com/primer-on-python-decorators/) (darmowy)
